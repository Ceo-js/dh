Introdução a Banco de Dados

Bancos de dados aqui, bancos de dados ali, mas o que é um banco de dados?

Podemos dizer que é um conjunto de dados pertencentes a um mesmo contexto organizado para uma finalidade específica. Um banco de dados nos permite:

  • Armazenar (adicionar, modificar, excluir) dados

  • Acessar os dados

  • Manipular dados e combiná-los

  • Analisar dados

  • Entre outras coisas

Existem dois tipos de bancos de dados: relacionais e não relacionais. Em particular, aprenderemos sobre um banco de dados relacional, pois nos permite entender em profundidade como os bancos de dados funcionam.

Por sua vez, dentro do mundo dos bancos de dados relacionais, existem várias opções pelas quais podemos escolher. No momento, veremos MySQL, já que é um dos bancos de dados mais populares do mundo e também é fácil de usar.

Vamos ver o que esse banco de dados pode fazer por nós.

Introdução a Banco de Dados

Os bancos de dados são hospedados em um servidor diferente da nossa aplicação web, ou seja, a porta em que vivem é diferente. Isso ocorre desta forma porque um banco de dados nem sempre é consumido por uma aplicação web, pois poderia ser facilmente consumido por outro tipo de serviço.

Ao instalar o MySQL no computador e iniciar o servidor, ele geralmente é habilitado na porta 3306. No entanto, nem sempre é esse o caso. Dependendo de como o MySQL foi instalado, a porta pode ser diferente.

Algo que vale a pena esclarecer é que um banco de dados não é em si um arquivo que podemos manipular, mas sim um conjunto de dados que se relacionam entre si e que fazem parte de um todo.

Tabelas

Tabelas

Certamente, em algum momento, já tivemos contato com uma planilha do Excel. Ela é composta por folhas, e em cada folha encontramos linhas, colunas e células.

Bem, uma tabela dentro de um banco de dados pode ser comparada a uma planilha em um arquivo Excel. Como qualquer tabela que se respeite, possui colunas e cada coluna tem a finalidade de definir um dado particular que será armazenado na referida tabela.

Ao criar uma tabela em nosso banco de dados, devemos ter em mente as informações que desejamos salvar e cuidar para que haja correlação. Por exemplo, se quisermos criar uma tabela para armazenar os dados pessoais de uma pessoa, o mais seguro é que a referida tabela tenha as colunas: nome, email, idade e senha, entre outros. E não seria muito lógico que a coluna "preço" existisse nesta tabela, pois esses dados não têm uma relação direta com as informações mencionadas acima.

Concluindo, uma tabela em um banco de dados busca armazenar as informações que fazem parte de um conjunto de dados. E busca, por sua vez, que todos os registros presentes na referida tabela tenham a mesma estrutura de dados, garantindo assim sua homogeneidade.

Constraints

Constraints

A tradução literal para o português da palavra constraint é: RESTRIÇÃO.

As constraint ou restrições dentro de uma tabela de banco de dados nos permitem gerar um esquema de informações muito mais estável e funcional aos olhos de nossa aplicação.

Por exemplo, uma das restrições mais comuns em uma tabela de pessoas é que as informações armazenadas na coluna de e-mail NÃO PODEM ser repetidas. Algo lógico, certo? Do contrário, o que aconteceria, por exemplo, se várias pessoas pudessem se cadastrar no Instagram com o mesmo e-mail? Seria um tanto complexo identificar quem é quem, não acham?

É por isso que as constraints são tão importantes. Elas nos permitem definir regras que devem ser atendidas para que os dados possam ser armazenados, onde estarão intimamente ligados à nossa lógica de negócios.

Foreign Key

Foreign Key ( Chave estrangeira )

Foreign key, também conhecida como FK ou chave estrangeira, é um tipo de constraint (restrição) que pode ser atribuída a uma coluna de uma tabela. Isso permite que essa coluna armazene como valor um dado particular que se refere aos dados armazenados em uma coluna de outra tabela, daí seu nome: "estrangeiro", por ser "de fora".

As chaves estrangeiras ou FKs permitem relacionar tabelas entre si e o objetivo desta particularidade é conceder maior desempenho na busca de algum dado particular.

Vamos ver um pouco mais sobre esse importante conceito então.

O que vamos ver nessa aula

Até agora, te apresentamos ao mundo dos bancos de dados relacionais, fazendo as coisas na mão com MySQL. Agora, é chegado o momento de usar todos esses conceitos que você adquiriu e implementá-los diretamente em um projeto com Express.

No entanto, para que o trabalho com MySQL e Express seja muito mais simples, vamos te mostrar uma ferramenta magnífica chamada Sequelize, que é um ORM para Express e que manipula bancos de dados relacionais.

Mas, o que é um ORM? Pois bem, um ORM é o que conhecemos como Object Relational Mapping, que é uma ferramenta de programação que permite "transformar" as tabelas de um banco de dados em uma série de entidades que simplificam as tarefas básicas de acesso aos dados. Essas entidades, quando recebidas em nosso programa, tem um formato de Objeto Literal, e como desde muito tempo a linguagem mais utilizada para trabalhar com banco de dados é o SQL, esses tipos de ferramenta já tem bastante suporte e são muito bem recebidas por parte das pessoas que trabalham na área de desenvolvimento e programação.

Para que fique um pouco mais fácil de entender o que faz um ORM, este diagrama talvez te ajude.

Em uma tabela do MySQL:

id name price stock
1 Nike shoes 123.45 35
2 Reebok sneakers 110.34 45

Depois de passar pelo ORM

[
{
id: 1,
name: 'Nike shoes',
price: 123.45,
stock: 35
},
{
id: 3,
name: 'Reebok sneakers',
price: 110.34,
stock: 45
}, ]

Assim, fica mais fácil de entender o que é um ORM. O que você vai ver nesta aula é:

Promises

Promises

Na programação, como na vida real, podemos fazer promessas, mas nos domínios de programação elas funcionam um pouco diferente.

Eles procuram executar um bloco de código que levará um certo tempo e então, com base na resposta obtida, poderemos fazer uma certa coisa.

Vamos pensar da seguinte forma:

Digamos que você está com fome e decide ir numa pizzaria nova no seu bairro, chegando lá nota que existe uma fila grande no balcão de atendimento, mesmo assim decide esperar.

Vários minutos se passam e sua vez chega. Um atendente bem mal-humorado te atende com uma cara de tédio e anota o seu pedido, imediatamente dirige-se até a cozinha e deixa a comanda no painel de comandas para então retornar ao seu posto. Você decide sentar em uma mesa para aguardar, mas o atendente pede que fique aguardando na fila até o seu pedido ficar pronto (impedindo o próximo cliente de fazer o pedido). Durante algum tempo o atendente fica parado lá olhando para sua cara e você, ainda sem entender direito, mantém-se em pé de forma desconfortável bloqueando as requisições que vêm após a sua.

O cheiro que sai da cozinha parece muito bom, entretanto, em uma breve olhada para trás você percebe que a fila só aumenta e os clientes parecem cada vez mais insatisfeitos com a demora para que apenas uma pizza seja despachada.

Assim seria trabalhar apenas com programação síncrona no javascript, já que o mesmo é single thread cada requisição não instantânea se tornaria um pesadelo para o cliente que lotaria a fila de espera de novas requisições. É por isso que existem as promises, conseguem enviar múltiplas requisições de forma que você, cliente, possa sair da fila e sentar em uma mesa até que o seu pedido fique pronto, permitindo que outros possam fazer os seus pedidos nesse meio tempo.

Por que aprender sobre promises?

As promises são uma ferramenta constantemente utilizada dentro da linguagem JavaScript e muitas tecnologias escritas com JavaScript as utilizam. Sequelize é uma delas, e é por isso que é tão importante entender o que são as promises.

Sem mais demoras, convidamos você a dar uma olhada no vídeo a seguir para saber um pouco mais sobre as promessas em JavaScript.

Normalmente, em linguagens de programação (quando o código é enviado para ser executado), o programa não continua a sequência até que o código enviado para ser executado termine e dê uma resposta. Nesse sentido, é bom saber que o JavaScript nos dá a possibilidade de modificar esta operação. Ou seja, mesmo que algum código seja enviado para execução, o programa pode continuar executando outros processos. Isto é chamado de Promises, situações que não sabemos quando serão cumpridas.
